## 2. APPLICATION LAYER

### Result pattern (`forzion.tech.Domain/Shared/`)
- `Result`/`Result<T>`/`Error` vivem em `forzion.tech.Domain.Shared` (movidos de Application — o DOMÍNIO usa Result para invariantes; Domain não pode depender de Application). `Result`: `IsSuccess`/`IsFailure`/`Error?`. Factories `Success()`, `Failure(Error)`, `Success<T>(v)`, `Failure<T>(Error)`. `Result<T>.Value` lança `InvalidOperationException` se acessado em falha.
- `Error(string Code, string Message, ErrorType Type=Business)` record com discriminador `ErrorType` {Business, Validation, NotFound, Conflict}; factories `Error.Business`/`Validation`/`NotFound`/`Conflict` (semântica em [specification-model]). Catálogos de domínio em `Domain/Shared/Errors/*.cs` (code `<agg>.<motivo>`). **`Error` SEMPRE via factory com `Type` explícito** — `new Error(...)` cru e `=> new(...)` (construtor implícito, default silencioso Business) são PROIBIDOS em catálogo/handler (gate `Tests/Architecture/ResultPatternConventionTests.cs`). `code` NUNCA muda numa reclassificação de `Type` (FE chaveia por `code`, nunca por status).
- **Regra de `ErrorType` (canônica)** — classifica pela natureza, não pelo texto:
  - **Validation (400)**: formato/tamanho/obrigatório/range de input, `*_id_invalido` (guard de ID = "menos errado" Validation). Ex.: `nome_obrigatorio`, `competencia_invalida`, `documento_invalido`.
  - **Conflict (409)**: transição de estado / já-X / duplicado / em-uso. `ja_*`, `apenas_*`, `nao_aguardando_*`, `transicao_*_invalida`, `*_duplicado`, `*_em_uso`, `possui_alunos`, `mesmo_plano`, `email_em_uso`, `offboarding_necessario`, `modo_inalterado`.
  - **NotFound (404)**: `*_nao_encontrad*`.
  - **Business (422)**: negação de POLÍTICA (não de estado nem input). Ex.: `cooldown_*`, `EliteIndisponivel`, `nao_disponivel`, `nao_pertence_treinador`, `plano_treinador_inativo`, `configure_stripe_primeiro`, `sem_onboarding`, `nao_pode_renovar_*`, `plano.inativo`. **DEFAULT do record (`new(...)`) = Business** — daí a proibição de construtor implícito.
- **Invariante quebrada (estado impossível, não erro de negócio) = `EstadoInconsistenteException` → 500** (NÃO `DomainException`/422 nem `Result.Failure`). Ex.: registro autenticado/pós-token-válido inexistente, `TipoConta` fora do enum. Mensagem diagnóstica fica server-side (`detail` genérico ao cliente, `code` ausente — ver §4). Gate proíbe `throw new DomainException` em `Application/UseCases/**`.
- Mapeamento HTTP no Api: `ResultExtensions.ToProblemResult()` faz switch em `Error.Type` → NotFound:404, Conflict:409, Validation:400, **default (Business):422**; `Results.Problem(detail=Error.Message, statusCode, title, extensions["code"]=Error.Code)`. O `title` é pt-BR por status (404→"Não encontrado.", 409→"Conflito.", 400→"Requisição inválida.", default→"Não foi possível processar."). Endpoints checam `result.IsFailure` e chamam `ToProblemResult()` (status já derivado do Type). O `cobrar`/renovar plano de treinador, no downgrade p/ Free, retorna `Result.Success` com flag `IniciarPagamentoPlanoResponse.AssinaturaEncerrada=true` (endpoint → 200 com mensagem, cron conta como não-falha) — NÃO há mais o code-sentinela `plano_free_assinatura_cancelada` (removido: `Result.Failure` jamais sinaliza sucesso).
- **Title pt-BR sempre** (regra): toda ProblemDetails que chega ao cliente tem `Title` em pt-BR. Três fontes: (1) path Result via `ToProblemResult` (acima); (2) `GlobalExceptionHandler` (títulos pt-BR por status); (3) erros auto-gerados pelo framework (binding/415/405/JSON malformado) via `CustomizeProblemDetails` + `UseStatusCodePages` (§ProblemDetails de framework). Mensagens exibíveis de domínio NÃO vazam termo interno (nome de enum/método/tipo PascalCase) — catálogo em `Domain/Shared/Errors`.
- POLÍTICA DE ERRO (regra de arquitetura — erro de NEGÓCIO usa Result; demais erros usam exception):
  1. DOMÍNIO retorna `Result`/`Result<T>` para toda invariante de negócio (NUNCA lança `DomainException` p/ regra de negócio).
  2. HANDLERS retornam `Result`/`Result<T>` (`Task<Result>`/`Task<Result<TResponse>>`) e propagam a falha de negócio do domínio direto (`return Result.Failure(...)`/`Result.Failure<T>(...)`). Não há mais handler que re-lança `DomainException` p/ traduzir falha de negócio. Endpoints desembrulham: `if (result.IsFailure) return result.ToProblemResult();` (422) senão `Results.Ok/Created(result.Value)`.
  3. EXCEPTION só p/ NÃO-negócio: control-flow cross-cutting (lookup miss `*NaoEncontradoException`, authz `AcessoNegadoException`, gates de login `EmailNaoVerificado`/`CredenciaisInvalidas`/`Treinador*`, conflito `EmailJaCadastrado`/`AlunoJaVinculado`, `LimiteAlunosAtingido`/`AlunoInativo`) + infra/programação (`ValidationException` da FluentValidation, `ArgumentNullException.ThrowIfNull`). Lançada no handler/contexto e mapeada pelo `GlobalExceptionHandler` — **mapa exceção→status HTTP é canônico no §4** (não reproduzido aqui).

### Use cases / handlers (CQRS-like)
Um handler por use case, organizado em `Application/UseCases/<Area>/<UseCase>/`. Pasta típica: `<X>Command.cs`/`<X>Query.cs` + `<X>Handler.cs` (+ `<X>Validator.cs`, `<X>Response.cs` opcionais). Convenções:
- Handler = classe com ctor primário (injeção), método público `virtual Task[<...>] HandleAsync(command, ct)` (virtual → mockável/override em testes). Guard `ArgumentNullException.ThrowIfNull(command)`; padrão comum: `HandleAsync` faz guard + delega a `HandleAsyncCore` privado.
- `TimeProvider` SEMPRE injetado para tempo (nunca `DateTime.UtcNow` direto na Application; testes injetam `FakeTimeProvider`).
- Persistência via interfaces de repositório + `IUnitOfWork.CommitAsync`. Operações com race condition usam `IDbContextTransactionProvider.BeginTransactionAsync(IsolationLevel.Serializable, ct)` (ex.: `AprovarVinculo`, `GerarCobrancaMensal`).
- Registro MANUAL no DI (`Api/Extensions/DependencyInjectionExtensions.AddApplicationHandlers`), todos `AddScoped<THandler>()`. NÃO há mediator/auto-scan de handlers.

### FluentValidation
- Validators `AbstractValidator<TCommand>` colocalizados na pasta do use case (ex.: `LoginCommandValidator`).
- Auto-descobertos: `services.AddValidatorsFromAssembly(typeof(LoginHandler).Assembly)` em `AddApplicationHandlers`.

## 3. DOMAIN EVENT DISPATCH (mecânica) — cross-cutting CHAVE
`AppDbContext` implementa `IUnitOfWork` + `IDbContextTransactionProvider`. `CommitAsync` (`Infrastructure/Persistence/AppDbContext.cs`):
1. **Coleta** entidades `IHasDomainEvents` rastreadas pelo `ChangeTracker` com `DomainEvents.Count > 0` (snapshot da lista ANTES de salvar). Se `eventDispatcher` é null, lista vazia.
2. **Snapshot + clear ANTES do `SaveChanges`**: copia todos os eventos para uma lista flat e chama `ClearDomainEvents()`. RE-ENTRÂNCIA: handlers podem chamar `CommitAsync` de novo; limpar antes garante "cada evento dispara exatamente uma vez" (sem isso o commit aninhado re-coletaria — ex.: projeção `Assinante` inserida 2x → duplicate key).
3. **Enfileira efeitos duráveis**: para cada evento cujo tipo é durável (`OutboxDurabilityRegistry.EhDuravel`), adiciona uma linha `outbox_efeitos` (`tipo=evt:<FullName>`, payload = JSON do evento, chave de idempotência do registry, `proxima_tentativa = OcorridoEm`). Vai ao `ChangeTracker` ANTES do save → atomicidade.
4. **`SaveChangesAsync`** (persiste agregado + linhas outbox na MESMA transação EF).
5. **Dispatch in-memory** via `IDomainEventDispatcher.DispatchAsync(events, ct)`.

`DomainEventDispatcher` (`Infrastructure/Services/`): para cada evento, monta `IDomainEventHandler<TConcreteEvent>` via reflection (`MakeGenericType(evento.GetType())`) e AGENDA o processamento best-effort em ESCOPO DE FUNDO (PERF-05). `IDomainEventHandler<in T>` tem default interface method que faz cast `IDomainEvent → T`.
- **Best-effort off request-path** (PERF-05): `DispatchAsync` NÃO awaita os handlers — agenda `Task.Run` que cria um `IServiceScopeFactory.CreateScope()` NOVO, resolve os handlers daquele escopo e roda `HandleAsync` com o CT do app (`IHostApplicationLifetime.ApplicationStopping`, opcional — fallback `CancellationToken.None`). Escopo novo é obrigatório: o da request já foi disposto quando o trabalho roda (resolver do escopo da request daria `ObjectDisposedException` no `DbContext`). O round-trip de e-mail/WhatsApp (Resend/Meta, timeout 15s) sai da latência do `CommitAsync`. Falha de handler é isolada (logada, não propaga); cancelamento no shutdown é best-effort (perda aceitável — D3). Seam `protected virtual ExecutarEmBackground` permite teste determinístico (rodar inline) — E2E usa `DispatcherBestEffortSincrono`.
- **Partição durável**: o agendamento best-effort PULA handlers marcados duráveis no `OutboxDurabilityRegistry` (rodam no worker do outbox, §3.1). `DispatchDuravelAsync(evento)` (usado só pelo worker) faz o oposto: roda SÓ os handlers duráveis SÍNCRONOS, no escopo AMBIENTE do worker (mesma tx/`DbContext` — atomicidade do efeito) e PROPAGA exceção (→ retry). Guard de regressão: `HandlerDiRegistrationTests` fixa o conjunto de `OutboxDurabilityRegistry.ParesDuraveis()` — remover um par o derrubaria silenciosamente p/ background.
- **Múltiplos handlers por evento**: suportado (ex.: `PagamentoEmDisputaEvent` → e-mail treinador + alert; `AssinaturaAlunoCanceladaEvent` → e-mail aluno + e-mail treinador + WhatsApp; `VinculoAprovadoEvent` → e-mail + WhatsApp + criar assinatura [durável]).
- **Boundary transacional**: handlers DURÁVEIS são atômicos com o agregado (enfileirados no MESMO `SaveChanges`, processados pelo worker, §3.1). Handlers BEST-EFFORT rodam APÓS o `CommitAsync` retornar, em escopo/`DbContext` SEPARADO — NÃO participam mais da tx serializable do use case (ex.: `AprovarVinculo`). Consequência consciente (PERF-05): o efeito durável crítico está no outbox (atômico); a notificação best-effort é fire-and-forget pós-commit.
- Resolução de handlers é por escopo de request (todos `AddScoped`). Detalhe de catálogo de eventos/produtores em [specification-model].

## 3.1 OUTBOX DE EFEITOS DURÁVEIS (transacional) — cross-cutting
Efeitos que NÃO podem se perder (mutação de negócio crítica, chamada a API externa) vão por outbox: persistidos na mesma transação do agregado e processados por worker com retry. Substitui o best-effort do §3 para esses casos específicos (`specification-coding §1`).
- **Tabela** `outbox_efeitos` (`specification-db`): `tipo`, `payload jsonb`, `status` (`Pendente|Processando|Concluido|Falhou`), `tentativas`, `proxima_tentativa`, `ultimo_erro`, `chave_idempotencia` UNIQUE, `processado_em`. Entidade `OutboxEfeito` (factory `Criar` + transições com guard de máquina de estado).
- **Dois estilos de `tipo`**: `evt:<FullName>` = re-dispatch de domain-event durável (#10, mutação); `fx:<nome>` = `IOutboxEfeitoHandler` por tipo (#8, efeito externo, ex.: evidência de disputa Stripe).
- **Enfileiramento**: `evt:*` no `AppDbContext.CommitAsync` (§3 passo 3, automático para tipos no registry); `fx:*` via `IOutboxEnfileirador.Enfileirar(tipo, payload, chave)` chamado pela use case ANTES do `CommitAsync` (mesmo UnitOfWork). `OutboxEnfileirador` serializa + `IOutboxRepository.Enfileirar` (Add sem commit).
- **`OutboxDurabilityRegistry`** (singleton, `BuildOutboxDurabilityRegistry` no DI): pares `(evento × handler)` duráveis + extrator de chave de idempotência por evento. Granularidade por handler (não por evento) porque um evento pode ter 1 mutação durável + N notificações best-effort. Handlers duráveis hoje: emissão/cancelamento NFS-e, criar assinatura de aluno, e-mail de suporte e **e-mail CRÍTICO** (`EmailCriticoSolicitadoEvent`→`EmailCriticoSolicitadoEmailHandler`, chave `evt:EmailCritico:{Id}` — verificação/reset/MFA/troca, carga cifrada via DataProtection; canônico em [specification-email] §STACK).
- **Worker**: `OutboxProcessorService : BackgroundService` (`Api/Services`, host fino, escopo por ciclo) delega a `OutboxProcessor` (`Infrastructure/Services`, lógica testável). `OutboxProcessor.ProcessarLoteAsync`: abre transação, lê lote sob lease (`IOutboxRepository.ObterProcessaveisAsync` → `FOR UPDATE SKIP LOCKED`), por item `MarcarProcessando` → `OutboxDispatcher.DespacharAsync` → `MarcarConcluido`/falha; `SaveChanges` + commit na MESMA tx (mutação do handler + avanço de status atômicos; locks soltam no commit). NOTA: o `CommitAsync` aninhado dos handlers (ex.: NFS-e) NÃO fecha a tx do lease — o outbox usa `context.Database.BeginTransactionAsync` direto (não a via `_transacaoAtiva`), então a tx envolve o lote inteiro.
- **Janela de lock do lease** (OPS-02): o lock é POR-LINHA (`SKIP LOCKED` ⇒ só as linhas leasadas param, a fila não bloqueia) e bound por dois timeouts, sem coluna `lease_expires_at`/reaper dedicado: (a) `HttpClient.Timeout` dos clients de efeito (NFS-e 30s, Stripe 30s, Resend/WhatsApp 15s — `InfrastructureExtensions`) limita o dispatch; (b) `SET LOCAL idle_in_transaction_session_timeout = OutboxOptions.TimeoutTransacaoIdle` (60s, > maior HttpClient.Timeout p/ não abortar dispatch normal) emitido logo após `BeginTransaction` — reaper do Postgres que mata a sessão (e solta os row-locks) se o worker congelar entre `BeginTransaction` e `Commit`. Garantido em código (não depende do `idle_in_transaction_session_timeout` do servidor/Supabase). `lease_expires_at`+reaper fica como follow-up explícito SÓ se essa config não cobrir.
- **`OutboxDispatcher`**: roteia por prefixo — `evt:` → `ResolverTipoEvento` (restrito aos registrados) + desserializa + `DispatchDuravelAsync`; `fx:` → `IOutboxEfeitoHandler` cujo `Tipo` casa.
- **Retry** (`OutboxOptions`, bind `Outbox`): `MaxTentativas` (5), backoff exponencial `BackoffBase·2^tentativas` (base 1min), `LotePorCiclo`, `IntervaloPolling`. Esgotado → `Falhou` + `LogCritical`. Idempotência: índice único em `chave_idempotencia` (re-enfileiramento bloqueado) + guards nos handlers (`specification-coding`).
- **Classificação transiente↔permanente** (OPS-01, `OutboxErroClassifier`): exceção PERMANENTE (`JsonException`, `ArgumentException` — payload corrompido/argumento determinístico, re-tentar não muda o resultado) ⇒ `MarcarFalhouDefinitivo` na 1ª ocorrência (não queima as `MaxTentativas`). Sinal ausente (HTTP 5xx/timeout `HttpRequestException`, `InvalidOperationException`, etc.) ⇒ default TRANSIENTE: retry+backoff (fail-safe, não enterra efeito recuperável). `OperationCanceledException` de shutdown é re-lançada antes da classificação (rollback do lease, não conta tentativa).
- **Backoff bound** (OPS-02, aceito): teto = `BackoffBase·2^(MaxTentativas-1)` = 16min @ base 1min/Max 5; limitado pelo nº de tentativas, não por cap explícito. Só vira problema se `MaxTentativas` subir muito.
- **Limpeza + observabilidade**: `OutboxLimpezaService : BackgroundService` (`Api/Services`, cadência `OutboxOptions.IntervaloLimpeza`=1h, separada do polling do worker) → `OutboxProcessor.LimparConcluidosAsync` remove `Concluido` com `processado_em < agora-RetencaoConcluidos` (7d) via `ExecuteDeleteAsync` (`IOutboxRepository.LimparConcluidosAnterioresAsync`). Estado do outbox (contagem por status + amostras de `Falhou`) exposto no relatório de saúde — ver [specification-observability] §3.
- **DI** (`InfrastructureExtensions`): registry singleton; `IOutboxEnfileirador`, `OutboxDispatcher`, `OutboxProcessor` scoped; `OutboxOptions` bind; `AppDbContext` recebe o registry. `OutboxProcessorService` + `OutboxLimpezaService` em `AddHostedService` (fora de Test).

### Bootstrap

### Endpoints (Minimal APIs, `Api/Endpoints/`)
Registro via extensões `MapXxxEndpoints` agregadas em `RouteBuilderExtensions.MapApiEndpoints`: Auth, Admin, HealthReport, Treinador, AlunoArea, Aluno, Conta, Mfa (`/conta/mfa`), MfaLogin (`/auth/mfa`), StepUp (`/auth/step-up`), Exercicio, Treino, Pagamentos, Webhook, Suporte, Internal. Padrão:
- Agrupamento por recurso com `MapGroup("/...")` + `.WithTags(...)` + `.RequireAuthorization("<Policy>")` + `.RequireRateLimiting("<policy>")` + `.AddEndpointFilter<...>()`.
- Handlers injetados via `[FromServices]`; contexto do usuário via `[FromServices] IUserContext` (`userContext.PerfilId`); `CancellationToken` no fim. Retorno: `Results.Ok/NoContent/NotFound/Problem` ou `result.ToProblemResult()`.
- Ex.: `AlunoAreaEndpoints` (`/aluno`, policy Aluno, `PaginacaoFilter`), `PagamentosEndpoints` (grupos `/aluno/pagamentos` read e `/treinador/pagamentos` write + endpoints internos), `WebhookEndpoints` (`/webhooks/stripe`, `/webhooks/resend` — anônimos, rate `webhook`, body limitado a 64 KB via `LimitedStream`), `SuporteEndpoints` (`/suporte`, `RequireAuthorization` + rate `write`; `POST /mensagens` → 202 Accepted, identidade do remetente vem do token NÃO do payload; e-mail ao suporte despachado async via outbox durável).

### Auth / authorization
- JWT Bearer (`Api/Configuration/AuthenticationExtensions`): segredo `Auth:JwtSecret` (>= 32 bytes, senão throw no boot), `JwtIssuer`/`JwtAudience` default `forzion.tech`, `ClockSkew=0`, `MapInboundClaims=false`. `OnTokenValidated` checa blacklist via `ITokenRevogadoRepository.EstaRevogadoAsync(jti)` → `ctx.Fail` se revogado (logout). Eventos de diagnóstico só fora de produção.
- Policies por claim `tipo_conta`: `"SystemAdmin"`, `"Treinador"`, `"Aluno"`.
- **Tokens de escopo restrito (MFA)**: `JwtService.GerarTokenEscopo(conta, escopo, validade)` emite token com claim `scope` (`MfaScopes.ClaimType`), SEM `tipo_conta`/`perfil_id` — `mfa_pending` (login pendente, 5min) e `step_up` (5min). A **default policy** (`SetDefaultPolicy`) agora exige autenticado **E** ausência de `scope` → token scoped não acessa rota de negócio. Policies dedicadas `MfaPendente`/`MfaStepUp` (`MfaScopes.PolicyPendente`/`PolicyStepUp`) gateiam só `/auth/mfa/*` e step-up. Postura completa em [specification-security] §2.1.
- `JwtService` (Infrastructure): claims `conta_id`, `tipo_conta`, `perfil_id`, `nome`, `jti` (novo Guid) e `fam` (família de refresh, só quando != Empty); expiração = **access curto por papel** (`SessaoConfig.AccessMinutes`: `Auth:Sessao:<papel>:AccessMinutes` → `Auth:JwtExpirationMinutes` → default Admin 10 / demais 15).
- **Refresh / `POST /auth/refresh`** (anon, cookie httpOnly `refresh`, rate `auth`): `RenovarSessaoHandler` rotaciona via `IRefreshTokenService.RotacionarAsync` — hash SHA-256 do raw → busca token → valida família ativa + token não-usado + dentro do idle/absolute. **Reuse** (token já `UsadoEm`) ⇒ `RevogarFamilia(ReuseDetectado)` + `LogWarning` + commit + 401. Sucesso ⇒ marca usado→sucessor, emite par novo, resolve perfil (treinador inativo → 401). Falha sempre 401 genérico (não vaza causa). Lifetimes/janelas: `SessaoConfig` (`IdleWindow`/`AbsoluteWindow`, `Auth:Sessao:<papel>`). Cross-ref [specification-security] §2.
- `Api/Context/HttpUserContext` (`IUserContext`, scoped): lê claims `conta_id`/`tipo_conta`/`perfil_id`/`jti`/`exp` + `fam` (`FamiliaId`, `Guid.Empty` se ausente); parse inválido lança `AcessoNegadoException`. Helpers `IsSystemAdmin`/`IsTreinador`/`IsAluno`.

### MFA / step-up (endpoints + handlers)
Todos com rate `auth`. Postura/fatores em [specification-security] §2.1; modelo em [specification-model]; e-mail OTP em [specification-email].
- **Enroll/gestão** (`MfaEndpoints`, `/conta/mfa`, `RequireAuthorization`): `POST /totp/iniciar` (`IniciarEnrollTotpHandler` → segredo + URI otpauth) · `POST /totp/confirmar` (`ConfirmarEnrollTotpHandler` → habilita + 10 recovery codes) · `GET /status` (`ObterStatusMfaHandler`) · `POST /desabilitar` (`DesabilitarMfaHandler`, **step-up**) · `POST /recovery/regenerar` (`RegenerarRecoveryCodesHandler`, **step-up**).
- **Login 2º fator** (`MfaLoginEndpoints`, `/auth/mfa`, policy `MfaPendente`): `POST /verificar` (`CompletarLoginMfaHandler` — TOTP|recovery|e-mail; `LembrarDispositivo` grava `TrustedDevice`; emite par de sessão) · `POST /email/enviar` (`SolicitarCodigoLoginEmailHandler` → OTP por e-mail).
- **Step-up** (`StepUpEndpoints`, `/auth/step-up`, `RequireAuthorization`): `POST /iniciar` (`IniciarStepUpHandler` — TOTP se habilitado, senão e-mail OTP) · `POST /verificar` (`VerificarStepUpHandler` → token `step_up` 5min).
- **Login** (`LoginHandler`): senha OK + MFA habilitado ⇒ NÃO emite sessão; retorna token `mfa_pending`. Trusted device válido pula o 2º fator. **Troca de senha/e-mail** revogam todas as sessões (`MotivoRevogacaoFamilia.TrocaSenha`/`TrocaEmail`).

### Filters (`Api/Filters/`, `IEndpointFilter`)
- `PerfilIdRequiredFilter`: rejeita (401) requisições autenticadas sem claim `perfil_id` (`PerfilId == Guid.Empty`). Aplicado em nível de grupo.
- `RequerStepUpFilter`: exige header `X-Step-Up-Token` (`RequerStepUpFilter.Header`) com token `step_up` válido; ausente/inválido → **403** `code=step_up_requerido`. Protege ações sensíveis: desabilitar/regenerar MFA, troca de senha, troca de e-mail, payout Stripe e ações admin. Cross-ref [specification-security] §2.1.
- `PaginacaoFilter`: valida query `pagina >= 1` e `1 <= tamanhoPagina <= 100`; senão 400.
- `RequireAssinaturaAtivaFilterBase` (abstract, T8) — base compartilhada `IEndpointFilter` para filtros de inadimplência. **GETs liberados** (sem checar assinatura — preserva leitura/histórico em inadimplência; princípio LGPD/CDC). Implementações injetam `IUserContext`, delegam checagem a `EstaInadimplenteAsync(services, userContext, ct)` e retornam 403 ProblemDetails com `code=CodigoErro` (abstract) se inadimplente.
- `RequireAssinaturaAtivaFilter : RequireAssinaturaAtivaFilterBase` (transient): impl aluno — `code=ASSINATURA_INADIMPLENTE`. `TipoConta != Aluno` → pass-through. Consulta `IAlunoRepository` + `IAssinaturaAlunoRepository` (última não-Cancelada). Cross-ref [specification-stripe]/[specification-frontend].
- `RequireAssinaturaTreinadorAtivaFilter : RequireAssinaturaAtivaFilterBase` (transient): impl treinador — `code=ASSINATURA_TREINADOR_INADIMPLENTE`. `TipoConta != Treinador` → pass-through. Consulta `IAssinaturaTreinadorRepository.ObterAtualPorTreinadorAsync` (single-row); `Status==Inadimplente` → 403. Aplicado em endpoints de consumo de feature paga do treinador (lista em [specification-stripe] §Enforcement backend). Não-Treinador / sem assinatura passam direto.

### Middleware / error mapping
`GlobalExceptionHandler` (`IExceptionHandler`, RFC7807 ProblemDetails):
- `ValidationException` (FluentValidation) → 400 `ValidationProblemDetails` com erros agrupados por propriedade (camelCase).
- Mapa de exceções de domínio → HTTP: `CredenciaisInvalidasException`→401; `*NaoEncontradaException`→404; `AlunoInativoException`/`AcessoNegadoException`/`EmailNaoVerificadoException`/`TreinadorAguardandoAprovacaoException`/`TreinadorInativoException`/`TreinadorPagamentoPendenteException`→403; `EmailJaCadastradoException`/`AlunoJaVinculadoException`→409; `DomainException`→422; `EstadoInconsistenteException`→500 (invariante quebrada/estado impossível; `detail` genérico, `code` ausente, `ex.Message` diagnóstica NÃO vaza — só log Error); demais→500 (mensagem genérica). `EmailNaoVerificadoException` (`EMAIL_NAO_VERIFICADO`), `TreinadorAguardandoAprovacaoException` (`TREINADOR_AGUARDANDO_APROVACAO`), `TreinadorInativoException` (`TREINADOR_INATIVO`) e `TreinadorPagamentoPendenteException` (`TREINADOR_PAGAMENTO_PENDENTE`) adicionam `code` extension consumido pelo frontend (login).
- Log: >=500 Error; <500 Warning.
- `Results` de falha de negócio com `Result` pattern usam `ResultExtensions.ToProblemResult()` → 422.

### ProblemDetails de framework (`AddApiServices` + `UseApiConfiguration`)
- `AddProblemDetails(o => o.CustomizeProblemDetails = ...)` preenche `Title` pt-BR por `Status` a partir do mapa ÚNICO `ProblemDetailsTitulos.PtBr` (`Api/Extensions/`; cobre 400/404/405/409/415/422/500) **só quando ausente ou ainda no título default em inglês do framework**. INVARIANTE: nunca sobrescreve um `Title` pt-BR já posto pelo path Result/`GlobalExceptionHandler` — checagem `IsNullOrEmpty(Title) || Title == ReasonPhrases.GetReasonPhrase(status)` antes de atribuir (o default inglês é DERIVADO de `ReasonPhrases`, não uma lista mantida à mão). FONTE ÚNICA: `ResultExtensions.ToProblemResult` lê o mesmo `ProblemDetailsTitulos.PtBr` por status — título não duplica literal entre os dois caminhos.
- `app.UseStatusCodePages()` (após `UseExceptionHandler`) é o que faz erros de framework SEM corpo (binding/415/405/JSON malformado das Minimal APIs, que retornariam 4xx vazio) passarem pelo `IProblemDetailsService` e ganharem corpo RFC-7807 — logo o `CustomizeProblemDetails` se aplica a eles. `GlobalExceptionHandler` escreve JSON direto (não passa por `CustomizeProblemDetails`); `Results.Problem()` passa.

### Rate limiting (`AddApiServices`)
Políticas FixedWindow (rejeição 429). Em ambiente `Test` todas viram NoLimiter. Chave: `auth`/`internal`/`webhook` por IP; `read`/`write`/`mfa` por `sub` (autenticado) ou IP (fallback). `mfa` (verificação de 2º fator em `/auth/mfa/*` e `/auth/step-up/*`) é sub-keyed de propósito — trava brute-force do TOTP por-conta, imune a rotação de IP. Caps autoritativos (números) em [specification-security] §4.

### Endpoints internos (server-to-server)
Todos anônimos + rate `internal`. Autenticação por header `X-Internal-Key` comparado a `Internal:ApiKey` com **comparação de tempo constante** (`CryptographicOperations.FixedTimeEquals`, após checar igualdade de comprimento — evita `ArgumentException` e timing attack). Sem/divergente → 401.
- `POST /internal/processar-renovacoes`: renovação de assinaturas de ALUNO — itera em LOTES keyset (`ListarParaRenovarAsync(now, cursor, N)`, [specification-performance] §2) chamando `GerarCobrancaMensalHandler` por assinatura, conta processadas/falhas.
- `POST /internal/processar-renovacoes-treinador`: renovação de assinaturas de TREINADOR — mesmo loop keyset sobre `IAssinaturaTreinadorRepository.ListarParaRenovarAsync` + `GerarCobrancaPlanoTreinadorHandler`; sucesso com flag `AssinaturaEncerrada` = downgrade p/ Free (não conta como falha).
- `POST /internal/reconciliar-pagamentos`: body opcional `{ desdeUtc }` (default janela 7d); chama `ReconciliarPagamentosStripeHandler`. Cross-ref [specification-infrastructure] (cron/chamador).
- `POST /internal/processar-pre-avisos`: pré-aviso de renovação 3 dias antes para assinaturas de aluno — `DespacharPreAvisosAlunoHandler` emite `CobrancaProximaAlunoEvent` por assinatura na janela; dispatch de e-mail via `CobrancaProximaEmailAlunoHandler`. Idempotência: janela diária do cron limita 1× por `DataProximaCobranca` (re-dispatch no mesmo dia reenviaria e-mail — aceito, dano baixo).
- `POST /internal/processar-pre-avisos-treinador`: mesmo padrão para planos de treinador — `DespacharPreAvisosTreinadorHandler` + `CobrancaProximaTreinadorEvent` + `CobrancaProximaEmailTreinadorHandler`.
- `GET /internal/lgpd/contas-elegiveis`: lista ContaIds elegíveis para purga LGPD (5 anos pós-cancelamento) — `ListarContasElegivelPurgaLgpdHandler`.
- `DELETE /internal/lgpd/contas/{id}`: anonimiza conta individual — chama `AnonimizarContaHandler` com `RealizadoPorId=SistemaPurgaLgpdId` (identidade sintética do job, `11111111-1111-1111-1111-111111111111`), sem confirmação de senha (caminho não-self).

### Webhooks

## 5. INFRASTRUCTURE LAYER

### EF Core / persistência
- `AppDbContext` schema-agnostic (sem `HasDefaultSchema`; schema vem do `Search Path` da connection), `UseSnakeCaseNamingConvention`, `MigrationsHistoryTable("__EFMigrationsHistory")`. Configs por entidade em `Persistence/Configurations` (`ApplyConfigurationsFromAssembly`). 40 DbSets (38 `public` — inclui `DataProtectionKeys` da infra de DataProtection, mapeada pela lib via `IDataProtectionKeyContext` sem Configuration própria — + 2 `internal` `TreinoExercicio`/`ExecucaoExercicio`, composição); `SerieConfig` mapeada SEM DbSet (acessada por navegação); `DadosFiscais`/`EnderecoFiscal` são owned VOs em `Treinador` (sem DbSet). Contagem de tabelas EF/NON-EF/migrations: canônica em [specification-db §STACK] (ancoragem `Tests/Architecture/SpecInventoryTests` — quebra força atualizar a spec db). Cross-ref [specification-db] (schema/migrations/enums).
- Registrado scoped em `InfrastructureExtensions` montando `DbContextOptions` na hora e passando o `IDomainEventDispatcher`. `IUnitOfWork` e `IDbContextTransactionProvider` resolvem para o MESMO `AppDbContext` scoped.
- Repositórios (`Persistence/Repositories`): classe com ctor primário `(AppDbContext context)`, métodos async; leituras de listagem usam `AsNoTracking`; paginação `(IReadOnlyList<T> Items, int Total)` com `Skip/Take`. Todos `AddScoped` em `InfrastructureExtensions`.

### Integrações + gate Null* (`InfrastructureExtensions.AddInfrastructure`)
- **Email** (`IEmailService`): `Resend:ApiKey` presente → `ResendEmailService` (HttpClient "resend", timeout 15s); senão `NullEmailService`. SEMPRE embrulhado em `EnvironmentEmailDecorator`. Cross-ref [specification-email].
- **WhatsApp** (`IWhatsAppNotifier`): `WhatsApp:PhoneNumberId` + `WhatsApp:AccessToken` presentes → `MetaWhatsAppCloudNotifier` (HttpClient typed para Graph API `v21.0` default, timeout 15s); senão `NullWhatsAppNotifier` (no-op, warning no ctor).
- **Stripe** (`IStripeService`): NÃO tem variante Null — `StripeService` sempre registrado, mas `StripeSettings` tem `ValidateOnStart` (boot falha sem SecretKey/WebhookSecret). Idempotência: refund usa `refund-{guid:N}` (`StripeService.cs:31`); PaymentIntent usa idem-key determinística `cobr:{tipo}...:{minuto}` montada pelo caller (esquema canônico em [specification-concurrency §1/§7]). Cross-ref [specification-stripe].
- `TimeProvider.System` singleton; `IJwtService`/`IPasswordHasher` (BCrypt) scoped.

### Domain event handlers (registrados por `IDomainEventHandler<T>`)
Localização e categorias:
- `Infrastructure/Notifications/Email/`: handlers de e-mail (Treinador aprovado/reprovado/inativado, Vínculo aprovado, AssinaturaAluno criada/inadimplente/cancelada×2, Aluno registrado/inativado, Conta registrada, Pagamento criado/falhou/estornado/em disputa).
