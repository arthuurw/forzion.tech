## NGINX / ROTEAMENTO
- `nginx/nginx.conf` (bind-mount ro). `:80` → 301 https (+ `/.well-known/acme-challenge/` p/ ACME). `:443` TLS (Let's Encrypt).
- `location /webhooks/` → `http://backend:8080` (ANTES de `location /`). Necessário p/ headers `svix-*` crus (webhooks **Stripe** `/webhooks/stripe` E **Resend** `/webhooks/resend`). NÃO usar proxy SPA `/api/backend` p/ webhook (descarta headers). Ver [specification-email].
- `location /internal/` → `return 404` (ANTES de `location /`, F3/REQ-F3.2). Fecha a borda p/ os endpoints internos (`X-Internal-Key`): nenhuma request externa alcança `/internal/*`. Os crons de billing/fiscal/LGPD batem nesses endpoints por DENTRO da VM via `docker compose exec backend curl localhost:8080/internal/...` (§Workflows billing/cron), não pela borda.
- `location /` → `http://frontend:3000` (proxy + ws upgrade). Backend NÃO tem outra rota pública (só via Next `/api/backend/[...path]` e `/api/auth/*`).
- **X-Forwarded-* / ForwardedHeaders**: nginx envia `X-Forwarded-For`/`-Proto`; o backend confia via `app.UseForwardedHeaders` (Homolog/Production, `RouteBuilderExtensions.UseApiConfiguration`, antes de Auth/RateLimiter). `KnownIPNetworks`/`KnownProxies` limpos + `ForwardLimit=1` — confia no único hop porque o backend só é alcançável via nginx (rede docker isolada, sem porta publicada). Premissa de segurança: se o backend passar a ter porta exposta, isto vira vetor de spoofing de IP — restringir a `KnownIPNetworks` então. Afeta rate-limit (IP real do cliente) e scheme em reconstrução de URL. Ver [specification-security] §4.
- Server `pact.homologacao.forzion.tech:443` → `pact-broker:9292` (usa cert do domínio principal até `certbot --expand`).
- **X-Robots-Tag (homolog)**: server block `homologacao.forzion.tech:443` injeta `add_header X-Robots-Tag "noindex, nofollow" always;` — homolog é host público; evita indexação por buscadores. NÃO replicar no server de produção (forzion.tech, futuro). Defesa em profundidade com `robots.ts` env-gated no frontend. Ver [specification-security] §3 / [specification-seo] §4.1.

- **access_log em arquivo + fail2ban (edge brute-force)**: nginx loga acesso em `/var/log/nginx/access.log` (combined), exposto no host via bind-mount `/var/log/forzion-nginx:/var/log/nginx` (`docker-compose.homolog.yml`) — stdout do container (json-file) NÃO serve pro fail2ban tailar. `scripts/setup-vm.sh` instala fail2ban e copia `infra/fail2ban/{filter.d/forzion-nginx-auth.conf,jail.d/forzion.local}`. Jail `forzion-nginx-auth`: 401/429 em `/api/auth*` + 429 em `/api/backend/*` → ban iptables. Limiar conservador (maxretry 12 / findtime 10m / bantime 1h) por NAT corporativo. Mecânica/rationale em [specification-security] §4. **Ban não testado em runtime — validar na VM (`fail2ban-client status forzion-nginx-auth`).**

## FIREWALL (ufw)
- **Provisionamento**: `scripts/setup-firewall.sh` (idempotente, standalone) é a fonte canônica das regras: `default deny incoming` + `default allow outgoing` + allow `22/80/443` + `--force enable`. `setup-vm.sh` o invoca (guard `if [ -f /opt/forzion/app/scripts/setup-firewall.sh ]`, idiom dos blocos fail2ban/systemd). Como é standalone+idempotente, roda sozinho na VM viva (`ssh ubuntu@<IP> 'bash -s' < scripts/setup-firewall.sh`) sem re-executar o setup-vm inteiro.
- **Caveats das regras** (rationale aqui, não em comentário no script — regra 9): (a) os `allow` vêm ANTES do `enable` — o script chega via SSH:22; habilitar `deny incoming` sem liberar a 22 cortaria a própria sessão; (b) `--force` porque `ufw enable` é interativo (y/n) e `bash -s` non-interactive travaria; (c) 22 fica PÚBLICA de propósito (fechar pro tailnet é endgame manual — §DEPLOY MANUAL porta-22; nunca embutir lockout na provisão).
- **Deploy via tailnet segue OK**: `ufw allow 22/80/443` vale em TODAS as interfaces (inclusive `tailscale0`) → SSH:22 + Pact broker:443 pelo tailnet não exigem regra extra. fail2ban coexiste (chains iptables próprias). Tailscale: `default allow outgoing` cobre o WireGuard de saída; conexão direta nó-a-nó pode cair pro relay DERP sob ufw, mas o deploy não quebra (DERP é fallback transparente).
- **⚠️ GOTCHA ufw NÃO filtra portas publicadas pelo Docker** (CANÔNICO): Docker escreve regras próprias na chain `DOCKER-USER` (avaliada ANTES do `INPUT` do ufw) ⇒ container com `-p` no host fica acessível MESMO com `ufw deny incoming`. Hoje no homolog **só o nginx publica** (80/443) — backend/frontend/pact ficam na rede Docker interna (sem `-p` em `docker-compose.homolog.yml`) — então a exposição efetiva já é 80/443 + 22, e o ufw protege de fato APENAS a 22 (serviço de host) + portas de host futuras não-Docker. A frase "firewall 80/443-only" (§ISOLAMENTO) é garantida pelo **que está publicado**, não pelo ufw. Pra PROD co-locado: se publicar via Docker `-p` algo que deveria ser privado, ufw não segura → usar regra `DOCKER-USER` (ferramenta `ufw-docker`) ou bind `127.0.0.1:porta`.
- **Impacto de editar os scripts é ZERO na VM viva**: provisionamento é one-time e a VM homolog já rodou; mudança só vale em provisão FRESCA (VM nova / 2ª VPS DR / prod) OU rodando `setup-firewall.sh` à mão na VM.

## ISOLAMENTO-PRD-HMG (co-locar prd + homolog) [ALVO]
Prd NÃO existe ainda. Checklist de host/VM/rede PARA QUANDO subir prd na MESMA VPS (ponte de custo). Postura de DADOS/segredos: [specification-security §10] (project Supabase separado é obrigatório; Stripe LIVE isolado). Invariante de isolamento: build cache sem teto enche o disco da VM e derruba containers sob pressão — com prd co-locado SEM limites, cairia junto; daí os caps abaixo.
- **Compose project separado** (`-p prd` / `-p hmg`) → redes Docker isoladas, container de hmg NÃO alcança prd.
- **`mem_limit` + `cpu_shares` + `pids_limit`** por serviço — impede hmg de matar prd por fome de recurso (CPU/RAM/IO/disco). **VM = 2 vCPU / 7.8 GiB / swap 0** (medido 2026-07-02). Split **80/20 prd/hmg** via `cpu_shares` (PESO relativo, não teto: prd 1024, hmg 256 → 4:1 sob disputa, mas cada um bursta a 100% quando o outro ocioso — build-on-VM do hmg não trava). Alocação (nos 2 composes; `docker compose` honra os 3 keys legacy — provado por inspect `HostConfig`): prd `backend` 1536m / `frontend` 768m (shares 1024, pids 512); hmg `backend` **384m** / `frontend` **256m** (shares 256, pids 256 — PISO bootável, staging pode ser lento por decisão do usuário); pact `broker` 512m / `postgres` 384m (shares 256). Soma mem_limit ≈3.75g + OS/docker ~1g ≈4.9g/7.8g. **SWAP 4GB OBRIGATÓRIO antes de co-locar** (`fallocate`/`mkswap`/`swapon` + `/etc/fstab`, `vm.swappiness=10`): sem swap, spike (build-on-VM uncapped do hmg, GC .NET, tráfego) = OOM-kill duro na prd. `cpu_shares` é INERTE até prd coexistir (peso só conta sob disputa); `mem_limit` NÃO capa o buildkit (build-on-VM do hmg segue livre → daí o swap). **DIFERIDO** (bring-up co-location): pact EFÊMERO (só up durante CI, libera ~0.9GB) — exige nginx upstream do `pact-broker` via variável+`resolver 127.0.0.11` (hoje é upstream ESTÁTICO na `nginx.conf` L145 → broker down derruba o nginx) + `profiles:[pact]` no compose + CI `--profile pact up`/`stop` em volta dos jobs pact. Por ora pact fica residente capado.
- **NÃO buildar na VM de prd** — imagens de registry via `release-images.yml` (CI builda+pusha ao GHCR `ghcr.io/arthuurw/forzion/*`; VM só `pull`). Build-on-VM é a fonte de acúmulo de cache no disco; em prd é inaceitável. Pull de imagem PRIVADA exige `docker login ghcr.io` na VM (§ENV, PAT `read:packages`).
- **Sem `docker.sock`** montado em container de app (= RCE→host).
- Firewall: só **80/443 inbound**; resto via tailnet. ⚠️ ufw NÃO filtra portas Docker-publicadas (chain `DOCKER-USER` bypassa o `INPUT`) — o "80/443-only" é garantido pelo que cada compose `-p` publica, não pelo ufw; com prd co-locado, NÃO publicar serviço privado via `-p` (usar `127.0.0.1:` ou `ufw-docker`). Detalhe em §FIREWALL. `/internal/*` fechado na borda (`location /internal/ { return 404; }`, F3) — crons acessam por dentro da VM via SSH+`docker exec` (§Workflows billing/cron). Admin de hmg via tailnet.
- nginx: server blocks por domínio, TLS+HSTS no domínio de prd, zero rota de hmg vazando no vhost de prd; `X-Robots-Tag noindex` SÓ no server de hmg ([specification-security §3]).
- `.env` separados, **600 root-only**, valores ZERO compartilhados entre ambientes.
- Idealmente: **rootless Docker / user namespaces** ou users Linux distintos por ambiente (isolamento real entre envs).
- Disco: monitor + ALERTA de uso (acúmulo de cache passa silencioso até encher). `docker builder prune -f --reserved-space 5g` no deploy (ci.yml) + timer systemd semanal (`scripts/docker-prune.sh`, instalado por setup-vm) barram a recorrência do cache; log-cap (`x-logging`) barra log ilimitado.
- **GOTCHA store de disco = containerd, NÃO `/var/lib/docker`**: o Docker desta VM roda com **containerd image store** (`Storage Driver: overlayfs`). Build cache + image layers vivem em **`/var/lib/containerd`** (`io.containerd.snapshotter.v1.overlayfs/snapshots`), NÃO em `/var/lib/docker` (lá só buildkit legado + volumes). Ao investigar disco: `du -xh /var/lib/containerd` — medir o dir errado dá falso "Docker é pequeno". `docker builder prune --reserved-space` alcança esse store.
- **`--reserved-space` (ex-`--keep-storage`) é teto, não alvo**, fixado em **`5g`**: build-on-VM recompila o app a cada commit (layers de output não reutilizáveis) → teto baixo mantém os layers de restore reutilizáveis e dropa o output stale. Manter ci.yml e `scripts/docker-prune.sh` em sincronia.
- **Poda automática (timer systemd)**: `scripts/docker-prune.sh` (builder prune keep 5g + image prune + `journalctl --vacuum-time=14d`) roda via `forzion-docker-prune.timer` (semanal, `Persistent=true`). `--volumes` OMITIDO de propósito (preserva `pact_pgdata`/dados). Lógica no `.sh` (git-tracked, atualiza no `git pull` do deploy); units `.service`/`.timer` exigem re-cópia (setup-vm).
