## 1. EF CORE — ARMADILHAS
- **N+1** — `Include` (ou projeção `Select` p/ DTO) p/ carregar relação navegada em loop; NUNCA materializar e navegar por filho dentro de iteração. Caso óbvio do domínio: listar `treinos` + `treino_exercicios` + `treino_exercicio_series` (composição 3 níveis, CASCADE) — uma query com projeção, não N consultas por exercício. [disciplina; detectável por contar SQL em teste de integração — §5]
- **`AsSplitQuery` em LISTA com `Include` de coleção aninhada** (PERF-03) — `Include(Exercicios).ThenInclude(Series)` numa query de LISTA (múltiplos treinos) gera JOIN cartesiano: cada série multiplica as linhas de cada treino × exercício (explosão de linhas/bytes na wire). `AsSplitQuery()` quebra em 1 query por nível de coleção (sem cartesiano) ao custo de +N round-trips — vale em lista, NÃO em single-row. Aplicado: `TreinoRepository.ListarPorTreinadorAsync`/`ListarPorAlunoAsync`, `TreinoAlunoRepository.ListarDetalhesPorAlunoAsync`. `ObterPorIdAsync`/`ObterDetalheAsync*` (single-row, 1 treino) ficam single-query de propósito (sem explosão). Split exige `OrderBy` determinístico (`+ ThenBy(Id)`) p/ paginação consistente entre as queries. [gate: integração conta linhas/queries — §5/Docker]
- **`AsNoTracking` em leitura read-only** — query que NÃO vai mutar+`CommitAsync` não precisa de change-tracker. Default tracking desperdiça memória/CPU em listagem (read model `assinantes`, catálogos `grupos_musculares`/`exercicios` globais). [disciplina]
  - CAVEAT (auditoria 2026-06-14): só aplicar onde a entidade lida NÃO é mutada depois na mesma unit-of-work. Vários reads que "parecem" read-only alimentam caminho de escrita e DEVEM ficar trackeados: `AssinaturaTreinadorRepository.ObterAtual*`/`AssinaturaAlunoRepository.ObterAtual*` (consumidos por Cancelar/TrocarPlano que mutam+commitam), `PagamentoTreinador` reads de webhook/pendente (mutam+commitam), `VinculoTreinadorAlunoRepository.ObterAtivoPorAlunoAsync` (consumido por `AprovarVinculoHandler`: na troca de treinador faz `vinculoAtivo.Inativar()` + `CommitAsync` — apesar de 4 outros callers serem read-only, este 1 caminho de mutação trava o método em tracking). Aplicado de fato (PERF-01) só a `PagamentoTreinadorRepository.ObterPagoPorAssinatura` (lê escalares pós-commit) e `ExecucaoTreinoRepository.ListarPorAluno`. `AsNoTracking` num read de caminho de escrita = bug silencioso (mutação não persiste).
- **Projeção > entidade inteira** — `Select` só as colunas usadas; não materializar agregado completo p/ devolver 3 campos.
- **Filtro no banco, não em memória** — `Where` antes de `ToList()`; NUNCA `ToList().Where(...)` (puxa tabela inteira e filtra no app).
  - Aplicado (remediação 2026-06-24): `RequireAssinaturaAtivaFilter` (caminho QUENTE — toda rota protegida de aluno) e `ObterAssinaturaAlunoHandler` materializavam TODO o histórico de assinaturas do aluno (`ListarPorAlunoAsync`) só p/ pegar a atual via `.Where(!=Cancelada).MaxBy(DataInicio)`/`.OrderByDescending(CreatedAt).First()` em memória. Trocados por `AssinaturaAlunoRepository.ObterAtualPorAlunoAsync` (resolve single-row no DB). Equivalência provada por invariante `DataInicio≡CreatedAt` em `AssinaturaAluno`: ambos setados 1× em `Criar`, `private set`, nunca remutados → `MaxBy(DataInicio)` ≡ `OrderByDescending(CreatedAt).First`. `ListarPorAlunoAsync` (uncapped) NÃO virou morto: único caller restante = export LGPD (precisa do conjunto completo, §2).
  - EXCEÇÃO documentada — `ContaRepository.ListarTesteAsync` (admin test-data): o filtro de sufixo `@e2e.test` fica client-side porque `Conta.Email` é value-converted (`HasConversion e=>e.Value`) → `EndsWith`/`EF.Functions.ILike` NÃO traduzem pelo conversor (ILike exige `string`; sem shadow-column normalizada não há pushdown). Mitigado: `Select(Id, Email, CreatedAt)` projeta só 3 colunas (não hidrata `PasswordHash`/segredos MFA — corta IO + PII em memória); descarta não-`@e2e.test` no app. Tabela pequena (só contas de teste em hmg). NÃO "corrigir" p/ `Where` no banco sem antes adicionar coluna de domínio normalizada (migration).

## 2. PAGINAÇÃO
- **OBRIGATÓRIA + cap** em todo endpoint de coleção (`skip`/`take` com teto, ex. max 100). Sem cap = scan de tabela inteira sob crescimento (alunos por treinador, pagamentos por assinatura, execuções por aluno). [disciplina — propor gate: teste de contrato rejeita endpoint de lista sem paginação]
- Ordenação estável (coluna determinística, ex. `created_at`+`id`) p/ paginação consistente.
- **Teto `Take(N)` defensivo só em LISTA de exibição, NUNCA em scan de correção** (PERF-02, auditoria 2026-06-14): cap em lista de display protege contra crescimento, mas truncar um scan que precisa do conjunto COMPLETO é bug de dados/compliance. NÃO capar: export LGPD, offboarding (bulk-mutate de vínculos/assinaturas), anonimização — precisam de TODAS as linhas. Cap aplicado só nas projeções-DTO de display `TreinoAlunoRepository` (`ListarAtivosComNome*`, `ListarAtivosPorTreinoId`) via `const LimiteListagem=500`; reads que retornam entidade p/ mutação em lote (`ListarAtivosPorPar/PorTreinador`) ficam SEM cap de propósito.
- **Overload paginado dedicado quando o MESMO read serve display + scan de correção** (remediação 2026-06-24): `PagamentoRepository.ListarPorAssinaturaAlunoAsync` é compartilhado por 3 callers — display do histórico do aluno (`ListarPagamentosAssinaturaAlunoHandler`), cancelamento (`CancelarMinhaAssinaturaAluno`) e export LGPD (`ExportarDadosPessoais`). Os 2 últimos precisam do conjunto COMPLETO (§2 PERF-02), então NÃO capar o método compartilhado. Fix: novo overload `ListarPorAssinaturaAlunoPaginadoAsync(assinaturaId, pagina, tamanhoPagina) → (Items, Total)` (AsNoTracking, `OrderByDescending(CreatedAt).ThenByDescending(Id)`, Skip/Take + Count) só p/ o display; clamp `[1,100]`/default 20 no endpoint via `PaginationExtensions.ObterPaginacaoDoQuery` (padrão dos demais endpoints de coleção). Endpoint `GET /aluno/pagamentos/assinatura/{id}` passou a devolver `{items,total,pagina,tamanhoPagina}` (era array nu) — mudança de contrato refletida em `openapi.v1.json` + frontend `pagamento.ts`/páginas aluno. Método uncapped original INTOCADO p/ cancel+LGPD.
- **Loop de cron/billing processa em LOTES (keyset), não materializa o conjunto inteiro** — query de fundo que itera linhas com efeito por-linha (renovação = Stripe + commit por assinatura) pagina por keyset `OrderBy(Id) + Where(Id > cursor) + Take(N)` (uuid é ordenável no PG). Offset/`Skip` QUEBRA aqui: linha renovada com sucesso SAI do filtro (`DataProximaCobranca` avança) → `Skip` pularia não-processados; o cursor keyset avança mesmo em FALHA (não reprocessa no mesmo run — próximo cron pega — sem loop infinito). `ListarParaRenovarAsync` (aluno+treinador) segue isso (PERF-01). NÃO paginam (decisão consciente): pré-aviso (`ListarParaPreAvisoAsync` — janela de 1 dia, só dispatch de evento, sem efeito por-linha) e `ListarNaoCanceladasPorTreinador` (limitado às assinaturas de 1 treinador). [disciplina]

## 3. CONEXÃO / DBCONTEXT
- `DbContext` scoped (1 por request); NUNCA capturar em singleton/campo de hosted service sem escopo próprio. Worker de outbox / hosted services (`LimparTokensRevogadosService`) criam escopo por iteração.
- **Loop de billing cria escopo por iteração** (PERF-02) — os endpoints `/internal/processar-renovacoes*` iteram N assinaturas chamando um handler com efeito por-linha. Resolver o handler 1× no escopo da request reaproveita o MESMO `DbContext` em todas as iterações → o ChangeTracker acumula entidades de todas as assinaturas, `SaveChanges` vira O(n²) e a memória cresce com o lote. Fix: `using var scope = scopeFactory.CreateScope()` DENTRO do `foreach` + resolver o handler do `scope` → `DbContext` fresco por assinatura (ChangeTracker O(1), memória estável). A query keyset (`ListarParaRenovarAsync`, `AsNoTracking`) fica no escopo da request — não trackeia, não acumula. Padrão espelha `OutboxProcessorService`.
- **Mesma classe estendida aos batch loops de NFS-e e reconciliação Stripe** (PERF-09, remediação 2026-06-24) — `gerar-nfse-comissao` (`GerarNfseComissaoMensalHandler`), `ReconciliarNfseHandler` e `ReconciliarPagamentosStripeHandler` rodavam o `CommitAsync` por-linha sobre o `DbContext` da request → mesmo O(n²)/memória do billing. Fix idêntico: `IServiceScopeFactory.CreateScope()` por iteração, resolvendo o trabalho por-linha (repo+`IUnitOfWork`+outbox, ou o sub-handler `ProcessarWebhookStripeHandler`) do scope fresco; o cursor externo (keyset NFS-e ou janela de Stripe-events, detached/`AsNoTracking`) fica no escopo da request. **CAVEAT ARQUITETURAL**: nesses três o loop vive DENTRO do handler (Application), então o `IServiceScopeFactory` foi injetado na camada Application (1º service-locator dela) — divergência CONSCIENTE do padrão de renovações (scope no endpoint/Api). Alternativa pura (hoistar o loop pro endpoint, handler vira "processa 1 linha") = refactor maior, ADIADO. **CAVEAT NFS-e**: `NotaFiscalRepository.ListarPorStatusAsync` passou a `AsNoTracking` (cursor externo não pode acumular tracked — único caller é esse handler) e o handler RE-CARREGA a nota por id (`ObterPorIdAsync`) dentro do scope p/ mutar+commitar; a entidade do cursor é só leitura de varredura. Verificado por unit test (mock); a persistência real (DB Testcontainers) valida no CI.
- Pool de conexão PG (Supabase) respeita o limite do tier; vazamento = `DbContext`/conexão longo-vivo sem dispose. Host Supabase direto é IPv6-only ([specification-db] §DICAS) — não é questão de perf mas de conectividade.
- **Sizing do pool Npgsql** (PERF-08) — defaults `MinPoolSize=1`/`MaxPoolSize=20`/`ConnectionLifeTime=15s` (CANÔNICO + caveat pgbouncer/Session pooler em [specification-db §DICAS — Pool de conexões]). O scope-por-iteração do billing e dos batch NFS-e/reconciliação (PERF-02/PERF-09) toma/devolve 1 conexão por linha serialmente (loops de cron/internal, sem concorrência) → cabe no teto. Não inflar `MaxPoolSize` acima do limite do projeto Supabase.

## 4. ASYNC & CANCELAMENTO
- **Async all-the-way** — `async`/`await` ponta-a-ponta; NUNCA `.Result`/`.Wait()`/`.GetAwaiter().GetResult()` (deadlock / thread-pool starvation). [disciplina — propor analyzer]
- **`CancellationToken` propagado** — do endpoint Minimal API até a query EF (`ToListAsync(ct)`); request abortado pelo cliente não deve segurar conexão/CPU. [disciplina]
- **Timeout explícito em I/O externo** (Stripe/Resend/WhatsApp via `HttpClient`) — sem timeout = thread presa indefinida se o provider pendurar. Efeitos externos garantidos vão por outbox ([specification-concurrency] §2), que isola a latência do provider do request do usuário.
- **Cliente Stripe global com timeout/retries explícitos** (PERF-04) — o `HttpClient` default do Stripe.net tem timeout de **80s**, alto demais p/ o request-path. `StripeClientFactory.Construir` monta um `StripeClient` com `HttpClient.Timeout` configurável (`Stripe:TimeoutSegundos`, default 30) + `SystemNetHttpClient(maxNetworkRetries)` (`Stripe:MaxNetworkRetries`, default 2), setado em `StripeConfiguration.StripeClient` no startup (`AddInfrastructure`, só se `SecretKey` presente). Os serviços usam `new XService()` → consomem o cliente global; a `ApiKey` por chamada (`RequestOptions`) sobrepõe a do cliente. CT propagado por chamada (`...Async(ct)`).
  - **TRADE-OFF ACEITO — estado global de processo** (PERF-04, E6-i3, decisão do usuário 2026-06-18): `StripeConfiguration.StripeClient` é singleton process-wide e o factory relê config no startup. Single-tenant ⇒ funcional e correto. Limite: o `RequestOptions.ApiKey` por chamada sobrepõe SÓ a key — **timeout e maxNetworkRetries ficam fixos no cliente global**, NÃO sobreponíveis por chamada nem por tenant. 3 eixos: segurança = key isolada por chamada (sem vazamento entre contas); performance = 1 `HttpClient`/`SocketsHttpHandler` reusado (sem custo de reconstrução); usabilidade = config simples p/ o modelo single-tenant atual. Hardening futuro (SE multi-tenant com timeout/retry divergente por conta): injetar `StripeClient` via DI em vez do estático. Cross-ref [specification-stripe §COMPONENTES].
- **NUNCA chamar I/O externo com tx aberta** (PERF-01) — chamada HTTP (Stripe `Create PaymentIntent`) dentro de uma tx `Serializable` segura o lock pelo RTT do provider → serializa cobranças concorrentes e desperdiça conexão do pool. Mover o I/O p/ ANTES da tx; tx só persiste o resultado (curta). Idem-key determinística + retry de 40001 cobrem a corrida ([specification-concurrency] §1/§3, [specification-stripe §IDEMPOTÊNCIA DA COBRANÇA]). Padrão: `CriarPagamentoComIntentService`.

## 7. FETCH-CACHE CLIENT (corta carga no PG)
- Frontend usa TanStack Query v5 como camada de fetch-cache (fundação em `frontend/src/lib/query/`; detalhe em [specification-frontend §DADOS-CLIENT]). RELEVÂNCIA backend/DB: reads read-mostly cacheados no cliente NÃO re-batem o backend a cada navegação → menos pressão no Postgres (Supabase Free: shared CPU / 500MB). `refetchOnWindowFocus:false` global é CRÍTICO aqui — default `true` re-dispararia toda query ativa a cada foco de aba, bombardeando o DB.
- Migrados (etapa 2): catálogo `grupos-musculares` + `planos` (stale/gc 30min — admin-curado, muda ~semanalmente) e burst do dashboard admin (stale 60s). Trade-off de staleness aceito (catálogo é lista de filtro, não dado transacional); registrado em [specification-frontend].
- NÃO é budget hard — disciplina. Ao migrar mais reads, `gcTime >= staleTime` (senão o cache é coletado ao sair da página e o ganho cross-navegação some).
